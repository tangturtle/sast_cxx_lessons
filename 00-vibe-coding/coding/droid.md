# Factory Droid

## 概览（Overview）

Factory.ai 的 Droid 是面向软件工程的“AI 编程/自动化代理”系列（如 Code Droid、Knowledge Droid 等），目标是在给定任务/工单的前提下，自动完成从理解代码库、规划变更、实现修改、运行测试到提交 PR 的端到端闭环。官方宣称在 Terminal-Bench 等基准上具备领先表现，面向个人与企业提供托管平台与终端工作流。

注：产品迭代迅速，下述为对公开资料能力的汇总；具体计费、模型、授权与 IDE/CLI 形态以官网为准。

## 核心能力（Key Features）

- 端到端自动化：从“读取任务/Issue”→“规划/分解”→“修改代码/生成补丁”→“运行测试/构建”→“创建 PR/变更说明”。
- 多代理/工具链：结合检索、静态分析、构建/测试、lint、安全扫描等工具，支持多轮尝试与回退。
- 上下文/记忆：持久化项目知识、依赖关系与历史决策，提高跨任务的一致性。
- 低延迟交互：与终端/聊天集成，支持人机在环的审批/强制门禁（如测试通过率）。
- 可观测性：动作轨迹、输入输出、补丁与日志可追踪，便于审计与回溯。

## 典型工作流（Workflow）

1. 在平台或 CLI 中选择仓库与任务（或粘贴错误日志/构建失败信息）。
2. Droid 自动建立上下文并规划子任务；必要时请求权限（读/写/执行）。
3. 生成 patch/diff 与说明；用户审阅并批准后应用到分支。
4. 运行测试/构建；若失败则迭代修复或回退。
5. 创建 PR，附带变更摘要、风险与验证步骤。

## 优势（Strengths）

- 自动化程度高：适合重复性修复、依赖升级、批量重构与跨仓库变更。
- 可审计：以 patch/PR 为边界，便于代码评审与合规记录。
- 团队协作：结合权限模型/审计与质量门禁，贴近工程治理实践。

## 局限（Limitations）

- 需要严格的权限与环境隔离策略，避免误改/越权执行。
- 在复杂业务/遗留代码上仍需人类工程师参与需求澄清与验收。
- 真实效果依赖于模型、依赖缓存、测试覆盖率与平台配置质量。

- Git 平台（分支/PR）、CI/CD、Issue 追踪、告警/监控系统。
- 终端与 Web 界面双入口；可与团队现有工作流衔接。

## 对比（Comparison）

- vs Devin/端到端工程代理：定位相近；模型/工具链/质量门禁与落地形态各异。
- vs Aider/Cline（CLI）：后者更“人控”，强调交互式补丁；Droid 更偏“自治执行+审批”。
- vs IDE Copilot：Copilot 行内补全/IDE 体验强；Droid 更偏任务级闭环与自动化。
