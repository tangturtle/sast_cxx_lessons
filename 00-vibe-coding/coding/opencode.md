# OpenCode

概览 / Overview

- 定位：由 SST 团队推出的“面向终端的 AI 编程代理”，原生 TUI（Terminal UI），支持在命令行内编写、修改、运行与测试代码。
- 典型场景：在终端中对项目进行增量改造、执行测试并根据失败结果迭代修复、搭建脚手架与配置流水线。
- 适用人群：长期驻留终端的全栈/后端工程师、DevOps/SRE、偏好最少窗口切换的开发者。

Key Features（核心特性）

- Native TUI：响应式、可主题化的终端界面，聊天与执行上下文同屏呈现。
- 任务与迭代：根据目标逐步生成补丁、运行命令、读取日志并继续计划（需要用户确认敏感动作）。
- 多模型/可插拔：支持接入多家 LLM 提供者与本地/远程后端（具体以官方更新为准）。
- 项目上下文：索引当前仓库、按需要载入文件/目录、生成最小化补丁以便审阅与回滚。
- 开源与可定制：开源仓库（sst/opencode），可根据团队安全/网络策略自托管与扩展。

Strengths（优势）

- 终端原生：无需离开 CLI，即可完成“理解→修改→运行→再修改”的闭环。
- 工程可控：以补丁/最小变更为单位推进，便于 Code Review 与审计。
- 易集成：与现有脚本、CI、包管理与测试框架自然衔接。

Limitations（限制）

- 生态仍在演进：与传统 IDE 插件生态相比，第三方集成与模板相对较新。
- 权限与安全：需要谨慎配置命令执行权限与网络访问策略，建议在非生产环境先行验证。
- 模型依赖：体验取决于所接入模型能力、上下文窗口与成本预算。

Use Cases（典型用例）

- 失败驱动修复：运行测试→收集失败日志→生成补丁→复测。
- 配置生成：根据自然语言生成 Dockerfile、CI 配置、Lint/Test 脚本并本地验证。
- 渐进式重构：按模块分批次生成 PR 级补丁，持续集成与回归。

Pricing / Access（获取与定价）

- 开源项目：代码与说明见 GitHub（sst/opencode），模型/推理成本由提供方计费。
- 可本地/内网部署：支持按团队策略自定义代理、缓存、密钥与日志策略。

Integrations（集成）

- Shell/终端：bash、zsh、PowerShell。
- 工具链：Git、包管理器、测试框架、容器/脚本运行器（依环境而定）。
- 模型：OpenAI、Anthropic、Google、及其他可配置后端。

Comparison（对比要点）

- 对比 Aider：Aider 强于 Git 工作流与补丁审阅；OpenCode 强于原生 TUI 与终端闭环。
- 对比 Gemini CLI/Goose：OpenCode 更偏向“终端工作台”体验，Goose 强在多模型与可扩展，Gemini CLI 强在谷歌生态与 ReAct 工具集成。
- 对比 IDE 扩展：OpenCode 更适合“命令行驱动”的开发方式，IDE 扩展适合图形化编辑与多窗格交互。
